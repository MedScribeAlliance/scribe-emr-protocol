# Contributing to MedScribeAlliance

Thank you for your interest in contributing to the MedScribeAlliance Protocol! This document provides guidelines for contributing to the specification.

## Ways to Contribute

### 1. Report Issues

Found a problem or have a suggestion? Open an issue:

- **Bug reports**: Inconsistencies, errors, or unclear sections in the spec
- **Feature requests**: New capabilities or improvements
- **Questions**: Clarifications about the protocol

### 2. Submit Pull Requests

Improvements to the specification are welcome:

- Clarifications and corrections
- Additional examples
- New sections or capabilities
- Typo fixes

### 3. Participate in Discussions

- Join GitHub Discussions to share ideas
- Provide feedback on proposed changes
- Share implementation experiences

## Contribution Process

### For Specification Changes

1. **Open an Issue First**
   - Describe the proposed change
   - Explain the rationale
   - Wait for feedback from maintainers

2. **Fork and Branch**
   ```bash
   git clone https://github.com/your-username/medscribealliance.git
   cd medscribealliance
   git checkout -b feature/your-change
   ```

3. **Make Changes**
   - Follow the existing document structure
   - Use consistent formatting
   - Include examples where helpful

4. **Submit Pull Request**
   - Reference the related issue
   - Describe changes clearly
   - Be ready to iterate based on feedback

### For Examples and Schemas

1. Place examples in `/examples` directory
2. Place JSON schemas in `/schemas` directory
3. Ensure examples are valid and tested

## Style Guide

### Markdown

- Use ATX-style headers (`#`, `##`, `###`)
- Use fenced code blocks with language hints
- Use tables for structured data
- Include navigation links at bottom of each spec file

### Code Examples

- Provide examples in multiple languages where possible
- Use realistic but fictional data
- Include comments for clarity

### Terminology

- Use RFC 2119 keywords (MUST, SHOULD, MAY) consistently
- Define new terms in the Introduction
- Be consistent with existing terminology

## Review Process

1. All changes require at least one review
2. Spec changes may require working group discussion
3. Breaking changes require consensus

## Code of Conduct

- Be respectful and constructive
- Focus on the technical merits
- Welcome newcomers
